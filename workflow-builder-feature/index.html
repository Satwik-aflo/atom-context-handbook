Workflow Builder for company-owned scrutiny.

ATOM Workflow Builder lets each company define how strict or trusting its inspection-to-repair workflow should be. The MVP is a UI-only mockup of the builder experience; Version 1 turns that experience into company-wide workflow templates, approval gates, tasks, and audit history.

Product intent

A trust policy builder, not a fixed approval chain.

The core idea is that ATOM should not decide every company's scrutiny model for them. Some companies will want Admin, Site Manager, and Project Manager review before work moves forward. Others will trust an SME or Technician enough to remove approval gates and move faster. Workflow Builder gives that accountability to the company.

Company chooses scrutiny

Approval gates can be required, reduced, or omitted based on the company's trust in its people and process.

ATOM keeps traceability

Inspection evidence, Findings, and Work Orders remain traceable workflow records even when approval gates are skipped.

Configuration stays explicit

Template changes, owner transfers, and runtime actions are audited so flexibility does not become invisible change.

MVP

Prove the builder experience first.

UI-only release

The MVP should create a new frontend page named Workflow Builder. It should be additive, use static or mock workflow data, and avoid touching existing workflow behavior until the team has validated the user experience.

MVP rule: build the visual workflow-builder experience and Workflow Inbox mockup without changing backend workflow execution, permissions, Finding transitions, or Work Order transitions.

Decision MVP choice Reason
Scope New Workflow Builder page only Lets the team evaluate the core product UX without disturbing live workflows.
Builder model Full drawing canvas The visual map is the essence of the feature, not a decoration added later.
Execution No backend execution changes Existing Finding and Work Order behavior should remain untouched for MVP.
Approvers Role-based approvers only This avoids user-specific complexity while still representing the main workflow model.
Inbox Include basic Workflow Inbox Configurable workflows make ownership less obvious, so users need a clear attention queue.

MVP feature list

The smallest complete mockup.

Canvas

Workflow Canvas

A Frappe-inspired drawing surface with state nodes, action nodes, connecting edges, zoom/pan, and selectable workflow elements.

Inspector

Properties Panel

A right-side panel for editing the selected node or action: label, action owner role, approval gate, return path, and notes.

Templates

Template Header

Shows template name, status, last edited metadata, current version, and clear actions such as preview, duplicate draft, and activate mock.

Approvals

Approval Gate Editor

Lets the user set approval gates to zero, one, or many role-based approver requirements. The MVP can show the behavior as mocked UI.

Inbox

Workflow Inbox

A basic cross-entity queue for urgent workflow handoffs, pending approvals, blocked gates, and tasks assigned to the user's role.

Warnings

Validation Rail

A side rail that distinguishes activation-blocking errors from warnings the company can accept or dismiss later.

Default workflow sketch

A baseline to edit from, not a hard ceiling.

The mockup should start from ATOM's default workflow and make it easy to see what the company has changed. Existing records should keep their current workflow instance when a new template is activated later.

Version 1

Make the company-wide workflow real.

Company-wide

Version 1 turns the MVP surface into a real company-level Workflow Template system. It should still stay company-wide, with per-site overrides saved for Version 2.

One active template

Each company has one active Workflow Template in V1. Drafts can be created from the default, from active templates, or from older templates.

Activation affects new work

New workflow instances use the newly active template. In-flight Findings and Work Orders remain on the template version they started with.

Rollback by reactivation

Companies roll back by activating an older template for future records, not by migrating existing records mid-flight.

Version 1 feature list

The full company trust policy builder.

Area V1 features Product rule
Template lifecycle Default template, draft templates, active template, archived templates, version history, edit-from-active copy, rollback by activating previous version. The default is immutable. Active templates cannot be edited directly.
Workflow configuration ownership At least one active Workflow Configuration Owner per company, initially Admin. Eligible owners include Admin, Project Manager, and Site Manager. Owners can add another owner and remove themselves only after another owner exists. Super Admin can recover ownership.
Canvas authoring State nodes, action nodes, edges, side properties panel, simulation, changed-from-default summary, activation warning. The UI should use ATOM language: Workflow Template, Action Owner, Approval Gate, Workflow Task, Workflow Inbox.
Approval gates Zero-to-many approver requirements, parallel gates by default, role-based and user-based requirements, return path on rejection. Zero approvers means no approval is needed. One rejection stops the gate and follows the Return Path.
Action ownership Configurable action owners for Finding creation, Work Order creation, Work Order completion, verification, stop paths, and contextual actions. Any configured role can own an action. ATOM should warn on risky choices, not block company accountability.
Workflow Inbox My Tasks, role-pool tasks, pending approvals, blocked gates, overdue tasks, and Workflow Health View for configuration owners. The inbox is for cross-person urgency and workflow handoffs, not every user's personal priority list.
Notifications Inbox and in-app notifications first, configurable email for high-priority approvals. No arbitrary webhooks, server scripts, or customer-authored automation code in V1.
Audit history Audit template edits, owner transfers, activations, runtime actions, display labels, internal action keys, and template version applied. Flexibility must remain reviewable after the fact.
Reporting labels Custom display states mapped to stable System Workflow Rollups for dashboards and filters. Entity pages can show company labels. Cross-company reporting uses stable rollups.

Validation model

Block broken templates, warn on risky choices.

Blocks activation

Validation Errors

  • Missing start state.
  • Missing terminal path.
  • Unreachable nodes.
  • Broken action owner.
  • Impossible approval requirement.
  • Missing return path for a rejecting gate.
Warns only

Configuration Warnings

  • User-based approver requirements.
  • Repeated user assignments.
  • Approval gates with zero approvers.
  • Omitted final review gates.
  • Role-boundary overrides.

Activation warning: activating a template affects only new records. Existing Findings, Work Orders, and other in-flight records should continue on their current workflow instance.

Deferred scope

Keep the first release focused.

Version 2

Per-site workflow templates

V1 templates should be designed so they can be applied per Site later, but V1 should stay company-wide.

Version 2

SMS and text notifications

Text-message workflow notifications are useful, but should follow after inbox, in-app, and email paths.

Later

Template import and export

Examples include sharing a common workflow, moving a tested template between environments, and starting from example templates.

Later

Arbitrary transition automation

Customer-authored webhooks, scripts, or background automations should not be part of the V1 workflow builder.

Design guidance

What the mockup should communicate.

The mockup should feel like an operational tool for governance users, not a marketing page. The first screen should show the actual builder: canvas, selected-node properties, validation rail, and access to the Workflow Inbox.

Canvas first

Make the workflow map visible immediately. Users should understand the process shape before reading detailed fields.

Dense but calm

Use compact controls, predictable panels, and clear status markers suited to repeated administrative work.

Trust choices are visible

Show where the company has increased or reduced scrutiny, especially skipped gates and role-boundary overrides.
